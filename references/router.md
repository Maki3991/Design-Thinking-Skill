# DT Router

## 这份文档是什么

这份文档是整个 DT skill 的一级路由入口。

它的职责不是代替所有 lesson 内容，而是先回答两个问题：

1. 用户现在的问题落在哪个 DT 阶段？
2. 接下来应该读哪一个或哪几个 reference？

默认原则：

- 所有问题先读 `router.md`
- `DT1.md` 是导论补充 reference，不是每次都要读
- 非导论问题优先直接跳到对应阶段文件

---

## DT 整体主线

整套课可以先压成这一条主线：

`Understand -> Define -> POV -> HMW -> Ideate -> Prototype -> Test -> MVP -> Iteration`

翻译成人话：

1. 先理解人、情境和自己
2. 再定义真正的问题
3. 收敛出一个清晰视角
4. 把视角改写成可探索的设计提问
5. 大量发散方案
6. 做低成本原型
7. 测试关键假设
8. 跑最小闭环
9. 根据反馈继续迭代

---

## DT1 的职责

`DT1.md` 不是所有问题的默认背景文件。

它主要负责三类内容：

- DT 的大图景和导论
- DT 如何用于人生设计
- Lesson 1 独有概念
  - `grow zone`
  - `迷茫不是退步`
  - `diverge -> converge`
  - `understand 自己`

所以只有以下问题，才默认补读 `DT1.md`：

- `DT 是什么`
- `为什么要学 DT`
- `DT 和人生设计是什么关系`
- `什么是 grow zone`
- `为什么迷茫也是成长`
- `什么叫发散和收敛`
- `怎么理解自己`
- `整套课的大图景是什么`
- `怎么把 DT 用在人生选择上`

如果问题已经明确落在 `define / hmw / prototype / test` 这些单阶段方法里，默认不要带上 `DT1.md`。

## 课程路由总表

### 1. `intro/life-design`

核心职责：

- 解释 DT 是什么
- 解释为什么 DT 不只是产品方法，也适用于人生设计
- 解释 Lesson 1 独立概念

默认读取：

1. `router.md`
2. `DT1.md`

典型问题：

- `DT 是什么`
- `为什么人生选择会迷茫`
- `如何理解“迷茫不是退步”`
- `DT 为什么适合做人生设计`
- `整套课的大图景是什么`

### 2. `define`

核心职责：

- 不要太快解题，要先定义问题
- 处理表象、根因、场景、角色、流程、痛点
- 进入 `5 Whys / 5W1H / Interview / Empathy Map / Insight / POV`

默认读取：

1. `router.md`
2. `define-problem.md`

典型问题：

- `怎么找到真正的问题`
- `怎么区分现象和根因`
- `怎么做用户访谈`
- `怎么做共情`
- `怎么提炼 insight`
- `怎么写 POV`

### 3. `hmw/ideate`

核心职责：

- 从 `POV` 进入设计挑战
- 用 `HMW` 打开可能性空间
- 发散方案并做初步筛选

默认读取：

1. `router.md`
2. `hmw-ideate.md`

典型问题：

- `什么是 HMW`
- `怎么把问题改写成设计问题`
- `怎么做 brainstorming`
- `怎么从大量点子里初步筛选`
- `什么时候该发散，什么时候该收敛`

### 4. `prototype`

核心职责：

- 把想法变成低成本原型
- 区分原型、成品和测试工具
- 处理 `CFP / CEP / Funky / Dark Horse / Box / Vision Prototype`

默认读取：

1. `router.md`
2. `prototype.md`

典型问题：

- `什么是 prototype`
- `原型和成品有什么区别`
- `CFP / CEP 是什么`
- `怎么做低成本原型`
- `什么是 Funky / Dark Horse / Box / Vision Prototype`

### 5. `test/mvp`

核心职责：

- 测试关键假设
- 理解 `MVP / 最小闭环 / A/B Testing / iteration`
- 从局部验证走向闭环验证

默认读取：

1. `router.md`
2. `test-mvp.md`

典型问题：

- `什么是 test`
- `什么是 MVP`
- `prototype 和 MVP 的区别`
- `怎么做 A/B test`
- `怎么验证支付意愿 / 痛点假设 / 方案偏好`
- `怎么从 prototype 进入迭代`

## 按人生设计流程路由

### 阶段 1：我现在很迷茫，不知道自己要什么

默认读取：

1. `router.md`
2. `DT1.md`

要点：

- 人生没有标准答案
- 迷茫不是退步
- 先从 `Understand` 开始，而不是逼自己立刻决定

### 阶段 2：我知道自己难受，但不知道真正卡点是什么

默认读取：

1. `router.md`
2. `define-problem.md`

要点：

- 区分现象和根因
- 区分抱怨和痛点
- 用 `5 Whys / 5W1H / Empathy / POV` 定义问题

### 阶段 3：我已经大致知道问题，但不知道有哪些路可走

默认读取：

1. `router.md`
2. `hmw-ideate.md`

要点：

- 把问题转成 `How Might We`
- 先发散，不要太早否定
- 先做可能性空间，再初步收敛

### 阶段 4：我有几个可能方向，但不知道怎么试

默认读取：

1. `router.md`
2. `prototype.md`

要点：

- 人生里的 prototype 是小范围试生活、试工作、试角色
- 原型不是正式决定，而是低成本验证

### 阶段 5：我试了一些方向，但不知道怎么判断结果

默认读取：

1. `router.md`
2. `test-mvp.md`

要点：

- 看关键假设有没有被验证
- 看有没有最小闭环
- 看反馈来自哪里：行为、体验、结果、付费意愿、持续性

## 给 AI 的使用说明

如果你是 AI，并被用户 `@` 到这份文档，请按下面方式工作：

1. 所有问题先读 `router.md`。
2. 先判断问题属于哪个阶段，再决定要不要补读 `DT1.md`。
3. `DT1.md` 只在导论、人生设计或 Lesson 1 独立概念问题里默认加入。
4. 单阶段方法问题优先直读对应 reference，不要把 `DT1.md` 当成固定背景材料。
5. 如果问题跨多个阶段，至少综合 2 个章节，不要只引用单一文件。
6. 回答时优先输出：
   - 当前问题处在 DT 的哪个阶段
   - 当前阶段的核心任务是什么
   - 下一步应该进入哪一节的方法
